L1 load rules forced by the evidence

Each amber/red finding becomes an explicit, documented rule — not silent cleansing.

#FindingL1 rule
R13,285 duplicate (RETURNID, DATETIMEINTRANSIT) milestone rows (P6) Dedupe on (RETURNID, ts), keep DELIVERED-class status over others; log discard count as a DQ metric.
R20.6% of returns have a 3rd milestone, always DELIVERED (P7/P8) State ③ = first DELIVERED-class milestone; later DELIVERED rows retained as raw events, not state changes.
R313.3% of returns' first milestone is already DELIVERED (P8) When no IN_TRANSIT row exists, state ② is taken from the EVRI first-scan event; if neither, ② is null and the return is excluded from the SLA population (matches mart behaviour).
R41.1% of order lines have 2–6 return requests (P4) Introduce request_attempt_seq per order line; Return Rate counts an order line as returned once (latest completed attempt).
R55.7% of returns carry no TRACKINGID (P5) Valid state-① returns; they simply cannot join carrier streams. Surface as "no-label cohort" rather than dropping.
R69.3% of requests have no transit rows (P9) These are the Idle population — confirmed consistent with the mart's IdleReturnRequests concept; current-state derivation handles them natively.
R7CDC lag is batch-shaped: median 1.6d, p95 3.6d, max 3.8d; 0.4% of rows have negative lag, worst −24 days (P11) Incremental load watermarks on INSERTED_ON with a 7-day reprocessing window (catches late and amended rows via UPDATED_ON). "Data as of" on every serving surface is defined from MAX(INSERTED_ON), never from calendar yesterday. Negative-lag rows are loaded but tagged as a DQ anomaly cohort.

P13 blocker — bridge repair. ZZ ORDERNUMBER ≠ WMS ORDER_SERIAL_NUMBER as written. Diagnose formats first, then test candidate transformations:
/* P15a · what do the two keys actually look like? */
SEL TOP 20 ORDERNUMBER, ORDERLINEITEMNUMBER, WAREHOUSEREFERENCE
FROM PRODVM.ZZ_RETURN_REQUESTED WHERE DATETIMEREQUESTED GE DATE - 7;

SEL TOP 20 ORDER_SERIAL_NUMBER, ORDERLINE_NUMBER, RETURN_NUMBER, ACCOUNT_NUMBER, TRADING_CODE
FROM PRODVM.RETURN_ITEM WHERE RETURN_DATE GE DATE - 7;

/* P15b · candidate: WAREHOUSEREFERENCE as the bridge key */
SEL COUNT(*) AS zz_lines,
    SUM(CASE WHEN ri.RETURN_NUMBER IS NOT NULL THEN 1 ELSE 0 END) AS matched_via_whref
FROM PRODVM.ZZ_RETURN_REQUESTED r
LEFT JOIN PRODVM.RETURN_ITEM ri
       ON TRIM(r.WAREHOUSEREFERENCE) = TRIM(CAST(ri.RETURN_NUMBER AS VARCHAR(20)))
WHERE r.DATETIMEREQUESTED GE DATE - 150;
/* P16 · decode the WH event vocabulary (incl. 0097) */
SEL e.DESPATCH_EVENT_CODE, e.DESPATCH_EVENT_DESC, e.DESPATCH_EVENT_GROUP_CODE
FROM PRODVM.RETURN_EVENT_CODE e;

/* P17 · ORDERSTATUS values — portal-side state signal? */
SEL ORDERSTATUS, COUNT(*) AS lines_, COUNT(DISTINCT RETURNID) AS returns_
FROM PRODVM.ZZ_RETURN_REQUESTED
WHERE DATETIMEREQUESTED GE DATE - 150
GROUP BY 1 ORDER BY 2 DESC;
P11 verdict — freshness ceiling for the platform (D5). The lag between a customer raising a request and the row landing in PRODVM.ZZ_RETURN_REQUESTED is batch-shaped: 25.7% arrive within a day, 72% within two, p95 = 3.6 days, hard ceiling 3.8 days. Six recurring fractional-second signatures account for 99.6% of rows — roughly one bulk load per day. Consequence: the v1 daily-batch design is compatible with this source, but its honest freshness promise is "requested up to ~2 days ago, complete to ~4 days ago" — not "yesterday". The near-real-time target state cannot be met from the PRODVM path at all; it requires the PRODVMUPD intraday route (or an upstream feed) and should be scoped as such in Phase 3. The 188 negative-lag rows (insert timestamp before request timestamp, worst −24 days) suggest either amended request timestamps on re-requests or clock/timezone drift — worth a 5-minute eyeball of a few RETURNIDs, but too small to affect the design.
Where this leaves the semantic layer: states ①–③ are fully reconstructable today from ZZ sources with measured quality; state ④ and the entire D6 refund spec (issued · when · value) are present in the WMS tables and gated only on the P15 bridge repair and P16 decode. Once those two land, the L1 state-event model covers the complete flow — request to refund — without warehousedb access.
